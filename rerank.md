## 已遇到的问题与解决方案

### 问题 1：`rerank` 改坏 Chroma `documents` 二维结构

- 现象：
  下游读取 `documents[0]` 时，把完整 snippet 当成字符串逐字遍历，出现字符级结果。
- 原因：
  早期实现把 `results["documents"]` 从 `list[list[str]]` 改成了 `list[str]`。
- 解决方案：
  保持 Chroma 原结构不变，始终写回 `documents = [[...]]`。
- 当前状态：
  已修复。

### 问题 2：只重排 `documents`，没同步重排 `metadatas / distances`

- 现象：
  snippet、source、distance 对不上，文本与来源错位。
- 原因：
  早期排序时只处理了文本，没有把 metadata 和 distance 一起移动。
- 解决方案：
  把 `(doc, metadata, distance)` 绑定成一组一起排序，再整体写回。
- 当前状态：
  已修复。

### 问题 3：`rerank` 结果被后续 `distance` 排序覆盖

- 现象：
  看起来跑了 rerank，但最终返回顺序还是原始 distance。
- 原因：
  raw / rewrite 分支在后面又调用了按 distance 截断的逻辑。
- 解决方案：
  raw 分支直接返回 rerank 后顺序；rewrite 分支 merge 后也不再按 distance 二次重排。
- 当前状态：
  已修复。

### 问题 4：中文 query 分词弱，`keyword_hits` 区分度不够

- 现象：
  整句被当成一个 token，中文材料题几乎退化成 distance 排序。
- 原因：
  早期规则更接近英文思路，没有针对中文 query 做有效切分。
- 解决方案：
  引入 `jieba` 分词，并配合：
  - `STOP_WORDS` 去掉问法词和语气词
  - `PROTECTED_TERMS` 保留理赔领域关键词和否定词
  - `DOMAIN_TERMS` 提前注入领域词
- 当前状态：
  已接入，但区分度还不够强。

### 问题 5：即使 `recall_n` 提高后，仍有问题答不出来

- 现象：
  候选池增大后，仍有一批题返回“未在文档中找到答案”。
- 原因：
  问题已经不只是 recall 数量，更多是：
  - 题型排序不准
  - 文档类型加权缺失
  - `/ask` 不能很好整合多个 snippet
- 解决方案：
  当前先保留更大的候选池，并把重点转到题型感知排序与回答拼接。
- 当前状态：
  未彻底解决。

### 问题 6：清洗语料后效果有提升，但 `/ask` 仍有“未找到答案”

- 现象：
  `需要提交哪些材料`、`非约定医院就诊能赔吗` 已改善，但流程题和补件题仍弱。
- 原因：
  当前 prompt 太保守，且 rerank 还不具备“题型感知”。
- 解决方案：
  下一轮优先：
  - 先稳 `use_rewrite=false` 基线
  - 给 rerank 增加“题型 -> 文档类型”的加权
  - 再放宽 `/ask` 的跨 snippet 整合能力
- 当前状态：
  待下一轮修。

## Light Rerank

- 优化项名称：`规则式 Light Rerank`
- 为什么要做：
  向量 recall 已经能捞回一批相关片段，但 topN 顺序不稳定，常把“相关但不够直接”的 chunk 排在前面。
- 怎么验证：
  在当前清洗后的 `claim_kb_b` 上，对比同一批候选的：
  - `recall top3`：按 `distance` 升序理解
  - `rerank top3`：当前 `/search` 返回顺序
  并结合 `/ask` 是否从“未找到答案”变成“可部分回答”。
- 使用的样本：
  当前 10 个问题中的 4 个代表样例：
  - `需要提交哪些材料`
  - `非约定医院就诊能赔吗`
  - `怎么申请理赔`
  - `不够材料怎么办`
- before：
  仅依赖向量 recall / 按 distance 排序时，材料题、流程题、补件题经常混入“看起来相关但不够直接”的片段。
- after：
  应用当前 rerank 后，部分材料题和边界题的 top1 / top2 更贴近直接答案块。
- 副作用 / 局限：
  中文关键词区分度有限，当前 rerank 仍不能替代 recall、prompt 和 rewrite。
- 当前默认策略：
  当前规则式 rerank 已接入 `/search` 与 `/ask`，本轮评估统一使用 `use_rewrite=false`。
- 可直接面试复述句：
  我在 RAG 检索里补了一层规则式 Light Rerank，不是引入额外模型，而是先修正工程正确性，再用关键词、标题和距离做轻量排序，解决“召回有了但顺序不稳定”的问题，但它目前还解决不了流程题和补件题的题型误排。

## 下一步修复方向

### 下一轮修复覆盖

1. 流程题还不稳定
影响问题：`怎么申请理赔`、`线上申请和线下申请有什么区别`
残留现象：虽然已经能把流程类结果拉向 `05_sop_报案到结案`，但 `怎么申请理赔` 仍会被“准备材料”“补件或调查”这类后段步骤抢走，主流程入口没有稳定排到最前。
下一轮修法：继续细化流程子类型，把“入口流程 / 申请方式 / 提交申请 / 审核到账”拆开加权，优先顶出 `报案或通知`、`选择申请方式`、`提交申请` 这些核心步骤。

2. 材料题与补件题仍然会串台
影响问题：`住院要交啥`、`需要提交哪些材料`、`发票有了但费用清单没了怎么办`
残留现象：材料清单已经能上来，但补件场景、材料摘要、费用摘要仍会混排，导致 top3 不够干净。
下一轮修法：补 `material` 与 `supplement` 的题型区分，让 rerank 能优先识别“材料清单块”和“补件处理块”，减少两类问题之间的相互抢位。

3. 补件题已部分可答，但动作路径还不完整
影响问题：`不够材料怎么办`、`发票有了但费用清单没了怎么办`
残留现象：系统已经能回答“补齐材料”“回医院补打印费用清单”这类结论，但还不够稳定地组织成完整的处理步骤。
下一轮修法：继续保留场景 SOP 的优先级，同时增强 `/ask` 对多个补件 snippet 的整合能力，让回答从“结论型”变成“步骤型”。

4. 条件型 / 资格型问题仍缺专项支持
影响问题：`谁可以申请理赔`
残留现象：metadata 驱动后，`/search` 已能优先命中 `condition` 相关块，但真正直接回答“申请人资格”的证据片段还不够集中，`/ask` 仍偏保守。
下一轮修法：保留 `condition` 的 metadata 路由，在 rerank 里继续压低材料摘要块，优先顶出更直接的资格判断标题和规则片段。

5. 边界型与条件型混合题已有改善，但还不够精确
影响问题：`非约定医院就诊能赔吗`、`等待期内住院能申请理赔吗`
残留现象：边界型题在 metadata 路由后已经能优先命中 FAQ / 规则块，但 `等待期`、`医院范围`、`急诊例外` 还会互相混排，导致结论不够完整。
下一轮修法：保留 `boundary` 路由，在 rerank 里细分“等待期 / 医院范围 / 急诊例外 / 免责摘要”的优先级，让边界题更稳定命中最直接的规则块。

6. `/ask` 的多片段整合能力仍然是主要瓶颈
影响问题：几乎所有“部分可答”的题，尤其是 `怎么申请理赔`、`不够材料怎么办`、`等待期内住院能申请理赔吗`
残留现象：检索已经不再是“完全没有相关内容”，但回答经常只拼出一部分，或者保守地结束在“其余信息未明确”。
下一轮修法：继续优化 `/ask` prompt，让它在不编造的前提下更积极地整合多个 snippet，并优先输出“已确认信息 + 未明确部分”的结构化回答。

### 下一轮执行顺序

1. 先给 rerank 增加“题型 -> 文档类型 / 标题路径”的细粒度加权。
2. 再把 `condition / boundary / material / supplement` 这四类题型的摘要块和直接答案块继续拆开优先级。
3. 最后继续调 `/ask` prompt，让多片段整合后的回答更完整、更像最终用户答案。

## 本周整体结论

- 本周最有价值的优化：

  当前 rerank 仍然偏“关键词 + 标题 + 距离”的弱规则，对流程题和补件题缺少题型感知。
